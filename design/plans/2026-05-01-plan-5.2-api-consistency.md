**Files:**

- Modify: `src/keel/commands/code/rm.py`
- Modify: `src/keel/commands/archive.py`

The pattern in each:
```python
out.info(f"Removed: {path}")
out.result({"removed": str(path)}, human_text=f"Decision removed: {path}")
```

`out.info` goes to stderr; `out.result` goes to stdout. Users see the same fact twice. The `out.result` line carries both the JSON and the human text — that's enough.


Look for `out.info("...")` immediately followed by `out.result(..., human_text="...")` where the two strings convey the same info.

- [ ] **Step 2: Drop the redundant `out.info`**

In each file, delete the redundant `out.info` line. Keep the `out.result` (it has the JSON shape).

If a test asserts on the `out.info` content (in stderr), update the test to look in stdout (where `out.result`'s human_text goes). Run tests and fix any breakage.

## Task 5: Standardize `removed` JSON key shape across rm commands

**Files:**
- Modify: `src/keel/commands/decision/rm.py`
- Modify: `src/keel/commands/deliverable/rm.py`
- Modify: `src/keel/commands/code/rm.py`
- (verify others — `milestone rm`, `task rm` already use `{"removed": "<id>"}` which is correct)

Target shape: `{"removed": "<id-or-name>", "path": "<absolute-path-if-applicable>"}`. For `code rm`, additionally include `"remote"` and `"worktree_path"` since the entity has more parts. The KEY `removed` is always the human-recognizable name/id; `path` (if present) is the absolute filesystem path that was removed; extras are command-specific.

Current shapes:
- `decision rm`: `{"removed": "<full-path-as-str>"}` — make it `{"removed": "<slug>", "path": "<abs-path>"}`
- `deliverable rm`: `{"removed": "<path>"}` (verify) — make it `{"removed": "<deliverable-name>", "path": "<abs-path>"}`
- `code rm`: `{"removed_remote": ..., "removed_worktree": ...}` — make it `{"removed": "<worktree-name>", "remote": ..., "worktree_path": ...}`
