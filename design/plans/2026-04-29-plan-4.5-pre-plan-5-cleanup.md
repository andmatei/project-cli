> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Land the high-leverage cleanup items surfaced by the post-Plan-4 reviews — plugin-readiness scaffolding, the path/scope abstraction (kills 90+ duplicated literals), and the three highest-frequency UX issues — before starting Plan 5 (milestones + ticketing plugins).

**Architecture:** Mostly additive refactoring. **Tech Stack:** Same as Plans 1-4 — Python 3.11+, Typer, Rich, Pydantic v2, markdown-it-py, Jinja2, tomlkit, questionary, pytest, ruff.

## Pre-decided open questions

1. **Plugin discovery**: Python entry points (`importlib.metadata.entry_points`). Two groups will be defined: `keel.commands` (Typer subapps that plugins can register) and `keel.ticket_providers` (which Plan 5 will activate). Plan 4.5 sets up the discovery mechanism but only the `commands` group is wired now; `ticket_providers` is a stub.

2. **Manifest extension surface**: a single `extensions: dict[str, Any] = {}` field on `ProjectManifest` and `DeliverableManifest`. Plugins put their config under `[extensions.<namespace>.<provider>]` (e.g., `[extensions.ticketing.jira]`). Core stays `extra="forbid"` everywhere except this one field.

3. **Public API contract**: `keel.api` re-exports the stable surface. Anything outside `keel.api` and `keel.testing` is internal.

4. **UX deferrals**: 8 of the 10 UX findings are *not* fixed in 4.5 (they don't compound with Plan 5). They're recorded in the new decision file and will land as a future polish plan.

5. **Backwards compatibility for `extensions`**: existing `project.toml` / `deliverable.toml` files without an `extensions` table continue to load (Pydantic default empty dict). Round-tripping a manifest preserves the `extensions` block byte-equivalently when present (tomlkit ordering caveat: dicts may reorder keys, but content survives).

def save_project_manifest(path: Path, manifest: ProjectManifest) -> None:
    doc = tomlkit.document()
    doc["project"] = _dict_no_none(manifest.project.model_dump())
    if manifest.repos:
        repos_array = tomlkit.aot()
        for r in manifest.repos:

        doc["repos"] = repos_array

```python
"""Top-level Typer app and global flags."""
from __future__ import annotations
from importlib.metadata import entry_points
import typer
from keel import __version__

app = typer.Typer(
    name="keel",
    help="Manage the ~/projects/ workspace.",
    no_args_is_help=True,
    add_completion=True,
)


# Empty subapp pre-registered so plugins can extend it via entry points.
# Plan 5 will activate this with bundled providers.
ticketing_app = typer.Typer(
    name="ticketing",
    help="Ticketing system integration (Jira, GitHub Issues, etc.). Provided by plugins.",
    no_args_is_help=True,
)
app.add_typer(ticketing_app, name="ticketing")


def _version_callback(value: bool) -> None:
    if value:
        typer.echo(__version__)
        raise typer.Exit()


@app.callback()
def main(
    ctx: typer.Context,
    version: bool = typer.Option(
        False, "--version", callback=_version_callback, is_eager=True,
        help="Show version and exit.",
    ),
    quiet: bool = typer.Option(False, "-q", "--quiet", help="Suppress info logs."),
    verbose: bool = typer.Option(False, "-v", "--verbose", help="Verbose logs."),
) -> None:
    """keel: scope-driven development scaffolder for ~/projects/."""
    if quiet and verbose:
        raise typer.BadParameter("--quiet and --verbose are mutually exclusive.")
    ctx.obj = {"quiet": quiet, "verbose": verbose}


def _load_plugin_commands() -> None:
    """Discover and load command-group plugins via the `keel.commands` entry-point group.

    A plugin entry point should resolve to a callable that takes the keel `app`
    (or a subapp like `ticketing_app`) and registers its commands on it. Example:

        # In a plugin's pyproject.toml:
        [project.entry-points."keel.commands"]
        my_cmd = "my_pkg.cli:register"

        # In my_pkg/cli.py:
        def register(app: typer.Typer) -> None:
            app.command(name="my-cmd")(my_cmd_func)

    For ticketing-specific plugins, use `keel.ticket_providers` (Plan 5).
    """
    for ep in entry_points(group="keel.commands"):
        try:
            register = ep.load()
            register(app)
        except Exception as e:  # noqa: BLE001
            # Don't crash the whole CLI if a plugin fails to load.
            typer.echo(
                f"warning: failed to load plugin '{ep.name}': {e}",
                err=True,
            )


# Built-in commands — register first, then load plugins.
from keel.commands.new import cmd_new  # noqa: E402
app.command(name="new")(cmd_new)

from keel.commands.list import cmd_list  # noqa: E402
app.command(name="list")(cmd_list)

from keel.commands.show import cmd_show  # noqa: E402
app.command(name="show")(cmd_show)

from keel.commands.phase import cmd_phase  # noqa: E402
app.command(name="phase")(cmd_phase)

from keel.commands.validate import cmd_validate  # noqa: E402
app.command(name="validate")(cmd_validate)

from keel.commands.archive import cmd_archive  # noqa: E402
app.command(name="archive")(cmd_archive)

from keel.commands.rename import cmd_rename  # noqa: E402
app.command(name="rename")(cmd_rename)

from keel.commands.migrate import cmd_migrate  # noqa: E402
app.command(name="migrate")(cmd_migrate)

from keel.commands.completion import cmd_completion  # noqa: E402
app.command(name="completion")(cmd_completion)

from keel.commands.deliverable import app as deliverable_app  # noqa: E402
app.add_typer(deliverable_app, name="deliverable")

from keel.commands.decision import app as decision_app  # noqa: E402
app.add_typer(decision_app, name="decision")

from keel.commands.design import app as design_app  # noqa: E402

"""Project lifecycle phases — single source of truth.

Plan 5 may add a customisable lifecycle DSL; for now this module is the only
place phase names live. Other modules import from here.
"""
from __future__ import annotations

PHASES: list[str] = ["scoping", "designing", "poc", "implementing", "shipping", "done"]
DEFAULT_PHASE: str = PHASES[0]


def is_valid_phase(name: str) -> bool:
    return name in PHASES


def next_phase(current: str) -> str | None:
    """Return the next phase after `current`, or None if `current` is terminal."""
    if current not in PHASES:
        raise ValueError(f"unknown phase: {current!r}. Valid: {PHASES}")
    idx = PHASES.index(current)
    if idx + 1 >= len(PHASES):
        return None
    return PHASES[idx + 1]
```

## Summary

After Plans 1-4 shipped, three review agents (architecture, UX, plugin-readiness)
identified a punch list of items worth landing before Plan 5 (milestones + a
plugin system for ticketing integrations). Plan 4.5 implements the high-leverage
items; this decision records what changed and what's intentionally deferred.

## What changed

### Plugin-readiness scaffolding (M1)

- **`extensions: dict[str, Any]`** field added to `ProjectManifest` and
  `DeliverableManifest`. `extra="forbid"` is preserved on every other field.
  Plugins put their config under `[extensions.<namespace>.<plugin>]`, e.g.
  `[extensions.ticketing.jira]`. Round-tripped via `tomlkit`.
- **`keel.testing`** package now hosts `projects`, `make_project`,
  `make_deliverable`, `source_repo` fixtures. Plugin authors enable them with
  `pytest_plugins = ["keel.testing"]` in their conftest.
- **`keel.api`** module re-exports the public/stable surface. Anything outside
  `keel.api` and `keel.testing` is internal and may change between minor
  releases.
- **`app.py`** loads command plugins via `[project.entry-points."keel.commands"]`.
  An empty `ticketing` Typer subapp is pre-registered for Plan 5's bundled and
  third-party providers to extend.

### Path/scope abstraction (M2)

- **`Scope` dataclass** promoted from data-only to behavioral. Properties:
  `unit_dir`, `design_dir`, `manifest_path`, `phase_file`, `decisions_dir`.
- **`workspace.design_dir`, `manifest_path`, `phase_file`** helpers added.
- **17 command files swept** to use the new helpers. ~90 hardcoded
  `design/<file>` literals removed.
- **`keel.lifecycle`** module created hosting `PHASES`, `DEFAULT_PHASE`,
  `next_phase()`, `is_valid_phase()`. `phase.py` and `workspace.read_phase`
  import from there. Plan 5's customisable lifecycle DSL will plug into the
  same module.

### UX consistency (M3)

- Duplicate `Created project: ...` (stderr) line removed from `new`,
  `deliverable add`, `decision new`, `phase` (transition mode).
- `validate` summary line moved to stdout (was on stderr while the table was
  on stdout — broke pipelines).
- Fix-it hints appended to all "not found" errors. Examples:
  `keel show ghost` → `error: project not found: ghost\n  Hint: run 'keel list' ...`
- README section added documenting project-name resolution rules
  (positional vs `--project` vs CWD detection).

### Polish (M4)

- **`keel.errors.ErrorCode`** enum: standardises the ~30 `code=...` strings
  used in `out.error` calls. Stable contract for JSON consumers.
- `workspace.resolve_cli_scope` now routes errors through `Output.error`
  (was raw `typer.echo`, which lost the JSON `code` field).
- Function-local imports hoisted to module-level where no circular-import
  justification exists.

## What's intentionally deferred

These UX findings are real but don't compound with Plan 5; they'll land as a
future polish pass.

- **`phase` ↔ `show` panel consistency**: `phase` is plain text, `show` is a
  Rich panel. Cosmetic; doesn't affect Plan 5.
- **`design export` double-H1**: the export wrapper injects `# <project>` then
  the design.md begins with `# <project> — Design`. Fix in a polish pass.
- **`-y` flag semantics**: `keel new beta -y` still requires `-d "..."`. The
  flag means "skip confirmation prompts" not "default missing args". Document
  rather than redefine.
- **JSON schema unification across `list`/`show`/`phase`**: the shapes are
  inconsistent; unifying is its own design exercise (likely Plan 6).
- **Decision rm vs supersedes docs**: the help text could be sharper about
  the audit-trail tradeoff.
- **Phase transition discoverability**: `keel phase` (read mode) doesn't list
  valid next phases. Small UX gap; not blocking.
- **Markdown_edit unused helpers**: `section_exists`, `replace_section`,
  `remove_line_under_heading` are test-only. Decide later: delete or
  document as plugin-facing API.

## Consequences

- `keel.api` becomes the stable plugin contract. Bumping the minor version
  with breaking changes to `keel.api` requires the next major. Things outside
  `keel.api` are free to evolve.
- `Scope` is now opinionated (carries path-resolution behavior). Tests that
  construct `Scope` directly should keep working — the new properties are
  additive.
- The `extensions` manifest field is the only "open" field in the schema.
  Plugins MUST namespace their data; leaking config into top-level keys still
  fails with `extra="forbid"`.
- The `[project.entry-points."keel.commands"]` and
  `[project.entry-points."keel.ticket_providers"]` groups are now reserved.
  External plugins should target these.
