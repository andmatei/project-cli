# Design: data model and project layout redesign

**Status:** approved (brainstorming, 2026-05-05).
**Successor:** an implementation plan written via `superpowers:writing-plans`.

## Goal

Resolve four interlocking concerns surfaced after publishing keel-cli 0.0.1
and keel-jira 0.0.1:

1. The `design/` folder mixes machine-readable manifests with human-authored
   content. Wonky.
2. Tasks must belong to a milestone, but many users have no use for
   milestones on small projects.
3. Deliverables and projects use different schemas (`deliverable.toml` vs
   `project.toml`) despite documents asserting they're the same shape.
4. Ticketing payload construction is hardcoded in each provider plugin, so
   per-project customization (titles, labels, custom fields) requires forking
   the plugin.

The redesign keeps the existing public contract where possible (commands,
`keel.api` exports, plugin protocol entry-points) but reshapes the
on-disk layout, the `Milestone`/`Task` semantics, the manifest schemas, and
the `TicketProvider` protocol.

This is a 0.0.x→0.1.0 inflection. We absorb the breakage now, before users
accumulate.

## Non-goals

- **Auto-migration of arbitrary external workspaces.** The migrator will
  handle the four projects in this maintainer's workspace plus any cleanly-
  scaffolded keel project. Hand-edited oddities are out of scope.
- **Sidecar tickets** (writing local markdown tickets via a non-Jira
  provider). The architecture supports it via plugins; we don't ship one in
  this redesign.
- **Multi-provider ticketing** in one project (push to Jira AND GitHub
  simultaneously). The `[extensions.ticketing.provider]` field stays a
  single string. Multi-provider is a Plan 9+ topic.
- **Schema-versioned manifests** (declared `schema = 1` field). Recorded in
  `open-questions.md` for later.

## Four decisions

### 1. Project layout — Option D

```
<project>/
├── project.toml                  # manifest (renamed shape, see Decision 3)
├── README.md                     # auto-generated synopsis + links
├── scope.md                      # human-authored, at root
├── design.md                     # human-authored, at root
├── milestones.toml               # only present if milestones used
├── decisions/                    # one .md per decision (existing convention)
│   └── 2026-05-05-slug.md
├── plans/                        # implementation plans (optional)
├── specs/                        # design specs (optional, for big designs)
├── .keel/                        # tool-managed state, gitignorable
│   ├── phase                     # current phase (was design/.phase)
│   └── lifecycle.toml            # snapshot of bound lifecycle
├── deliverables/<name>/          # recursive — same shape (Decision 3)
│   └── (project.toml, scope.md, design.md, decisions/, .keel/, code-*/, ...)
└── code-<repo>/                  # git worktrees
```

**Why this shape.** Surfaced from a 5-agent review; converged from two
ecosystem researchers independently citing Cargo, dbt, Terraform, and
Obsidian. Every concern has its place:

- Manifests at root match Python convention (`pyproject.toml`).
- `.keel/` holds tool state, mirroring `.git`/`.terraform`/`.obsidian/`.
  Gitignore is one line.
- Human content (scope, design, decisions, plans) lives at the project
  root or in purpose-named subfolders, not under a wrapping `design/`
  folder.
- Recursion is genuine: a deliverable looks identical to a project at
  any depth.

**`README.md` template.** Auto-written by `keel new`:

```markdown
# {{ project.name }}

{{ project.description }}

**Lifecycle:** {{ lifecycle.name }} · **Phase:** {{ phase }}

## Design

- [Scope](scope.md) — boundaries and success criteria
- [Design](design.md) — living technical design
- [Decisions](decisions/) — one record per non-obvious choice
- [Plans](plans/) — implementation plans
{% if has_milestones %}- [Milestones](milestones.toml) — work breakdown{% endif %}

## Code

{% for repo in repos %}- `{{ repo.worktree }}/` — {{ repo.remote }}
{% endfor %}
```

The README is regenerated by `keel show --update-readme` if a user wants to
refresh it after phase changes. Otherwise it stays a creation-time snapshot.

**`.keel/lifecycle.lock.toml`.** A snapshot of the lifecycle bound at `keel new`
time. Today the lifecycle is re-resolved from
`<projects-dir>/.keel/lifecycles/<name>.toml` on every command. If the user
edits that file, every project using it changes silently. Snapshotting
makes lifecycle drift explicit and adds a `keel lifecycle update <project>`
command for opt-in refreshes. The `.lock.toml` suffix follows the Cargo /
uv / mise convention for "auto-managed, don't hand-edit, commit to VCS."

#### Five categories of state in a project

The split between `project.toml`, `milestones.toml`, `.keel/phase`, and
`.keel/lifecycle.lock.toml` follows a principled rule. Each file fits one of
five categories — different write-frequency, different ownership, different
"if I delete this, what happens" semantics:

| # | Category | Examples | Owner | Mutable | Lock-shaped |
|---|---|---|---|---|---|
| 1 | Declared identity | `project.toml` | User | Rarely | No |
| 2 | Declared work | `milestones.toml`, `scope.md`, `decisions/`, `plans/` | User | Yes (intentional) | No |
| 3 | Current state | `.keel/phase` | keel | Constantly | No |
| 4 | Locked snapshots | `.keel/lifecycle.lock.toml`; future `.keel/plugins.lock.toml` | keel | Only on explicit refresh | **Yes** |
| 5 | Derived caches | future `.keel/cache/` | keel | Regenerable | No |

Naming convention: anything in category 4 ends in `.lock.toml`. Plugin
authors writing their own lock data put it under
`.keel/plugins/<name>.lock.toml`. Surveyed tools (Cargo, uv, Poetry, mise,
Terraform) all use per-concern lockfiles rather than monolithic ones —
keel follows that pattern.

### 2. Hierarchy — implicit default milestone

The hierarchy stays four levels: project → deliverable → milestone → task.
Tasks always belong to a milestone, structurally, but users who don't want
milestones never see them.

**Rule.** When `keel task add` is invoked without `--milestone`, keel
auto-creates a single `default` milestone if it doesn't exist and adds
the task there. The default milestone is structurally identical to user-
defined milestones; the only special thing is that it's auto-created.

**UX consequences.**

- A new user runs `keel task add t1 --title "Set up"` with no milestone in
  mind. keel auto-creates `default` with title `"Tasks"` and `description = ""`.
- `keel show` and `keel task list` skip rendering the milestone hierarchy
  when the only milestone is `default` (UI-level — the data is unchanged).
- Promoting to explicit milestones: `keel milestone add m1 --title "Foundation"`
  and then `keel task move t1 --milestone m1`. The default milestone is
  auto-removed when its task count hits zero (no need to `keel milestone rm
  default`).

**Ticketing consequences.** Ticket payloads (Decision 4) reference
`task.milestone`. If `milestone.is_default = True`, plugins should omit
milestone references in their default templates. Users who want decoration
write the conditional themselves in their override.

### 3. Deliverables — same schema as projects, recursive

Drop `deliverable.toml`. Every nested unit uses `project.toml`. The schema
is identical at every level.

**What changes from today.**

- `DeliverableManifest` and `DeliverableMeta` Pydantic models are deleted.
- `ProjectManifest` and `ProjectMeta` get one new optional field:
  `shared_worktree: bool = False`. Today this is a `DeliverableMeta` field;
  any project (top-level or nested) can declare it.
- The `parent_project` field disappears. The relationship is path-derived:
  if `<projects>/foo/deliverables/bar/project.toml` exists, then `bar`'s
  parent is `foo`. No redundant declaration.
- `keel deliverable add` is now sugar for "create a project at
  `<parent>/deliverables/<name>/`." Internally calls the same scaffold
  path as `keel new`, with the parent's lifecycle/extensions inherited
  unless overridden.

**Inheritance rules.**

A nested project inherits these from its closest ancestor by default
(can be overridden per-level):

- `lifecycle` (from `[project] lifecycle = ...`)
- `[extensions.ticketing]` and per-plugin sub-blocks
- Any future plugin-defined inherited values

A nested project does NOT inherit:

- `[[repos]]` — each level declares its own (or none, if shared_worktree).
- Phase — each level has its own `.keel/phase`.
- Milestones — each level has its own `milestones.toml`.

Inheritance is "look up the chain until found." `keel show` of a
deliverable surfaces the inherited values explicitly so users can see
what's local vs inherited.

### 4. Ticketing — per-plugin templates, typed domain objects

Drop the idea of a core renderer. Each plugin owns its templates, including
defaults and the user-config schema.

**`TicketProvider` protocol — minimal, typed.**

```python
@runtime_checkable
class TicketProvider(Protocol):
    name: str

    def configure(self, config: dict) -> None:
        """Validate and accept the [extensions.ticketing.<name>] config."""

    def create_milestone(self, milestone: Milestone, scope: "Scope") -> Ticket:
        """Create a ticket from the typed Milestone. The plugin renders
        templates, looks up provider-specific config, builds the payload,
        and returns a Ticket with at minimum `id` and `url` populated."""

    def create_task(self, task: Task, scope: "Scope") -> Ticket:
        """Same shape. The plugin walks `task.milestone` to find the
        parent milestone's ticket_id if it needs the parent reference."""

    def transition(self, ticket_id: str, target_state: str) -> None: ...
    def fetch(self, ticket_id: str) -> Ticket: ...
    def link_url(self, ticket_id: str) -> str: ...
```

Notable changes from v0.0.1:

- `parent_id: str` parameter dropped from `create_milestone`/`create_task`.
  The plugin walks `scope.manifest.extensions.ticketing.<name>.parent_id`
  for milestones, and `task.milestone → milestone.ticket_id` for tasks.
- Plugins receive the full typed `Milestone`/`Task` keel domain objects.
  Templates can reference any field (`task.depends_on`, `task.branch`,
  `task.is_default_milestone_task`, etc.).
- Pre-rendered title/description strings are no longer passed in. The
  plugin renders.

**Per-plugin config shape.**

```toml
[extensions.ticketing]
provider = "jira"

[extensions.ticketing.jira]
url           = "https://acme.atlassian.net"
project_key   = "PROJ"
parent_id     = "PROJ-123"
labels        = ["keel", "{{ milestone.id }}"]
custom_fields = { "Sprint" = "{{ milestone.title }}" }
status_map    = { planned = "To Do", active = "In Progress", done = "Done", cancelled = "Cancelled" }

[extensions.ticketing.jira.templates]
task_summary          = "{{ task.title }}"
task_description      = "{{ task.description }}\n\n— keel: `{{ task.id }}`"
milestone_summary     = "{{ milestone.title }}"
milestone_description = "{{ milestone.description }}"
```

Defaults are plugin-specific. `keel-jira` ships sensible defaults that
work without any config beyond `url` + `project_key`.

**No shared rendering utility in keel-cli.** Each plugin picks its own
templating engine (Jinja2 / str.format / Handlebars / whatever) and
builds its context from the typed `Milestone`/`Task`/`Project`/`Scope`
objects the protocol already passes in. If two future plugins end up
writing identical Jinja boilerplate, that's the signal to extract a
shared helper — speculative shared utilities in core are out of scope.

**Extension points each plugin gets** (all optional, all default to no-op):

1. `DEFAULT_TEMPLATES`: class-level dict of plugin-specific defaults.
2. `template_context()`: returns extra Jinja variables namespaced under
   the plugin's name (`{{ jira.epic_url }}`).
3. `jinja_filters()`: returns custom Jinja filters (e.g. `{{ desc | adf }}`
   for keel-jira's markdown-to-ADF conversion).
4. `post_render(rendered)`: validate or transform after rendering; emit
   warnings for provider-incompatible content.

### Combined manifest example after all four decisions

`<project>/project.toml`:

```toml
[project]
name        = "alpha"
description = "Build a thing."
created     = 2026-05-05
lifecycle   = "default"

[[repos]]
remote        = "git@github.com:acme/foo.git"
worktree      = "code-foo"
branch_prefix = "me/alpha-foo"

[extensions.ticketing]
provider = "jira"

[extensions.ticketing.jira]
url         = "https://acme.atlassian.net"
project_key = "PROJ"
parent_id   = "PROJ-123"
labels      = ["keel"]
status_map  = { planned = "To Do", active = "In Progress", done = "Done", cancelled = "Cancelled" }
```

`<project>/milestones.toml` (only created when first `keel milestone add` or
`keel task add` runs):

```toml
[[milestones]]
id          = "default"      # auto-created on first `keel task add` without --milestone
title       = "Tasks"
status      = "active"
ticket_id   = null

[[tasks]]
id          = "t1"
milestone   = "default"
title       = "Set up"
status      = "planned"
ticket_id   = null
```

`<project>/.keel/phase`:

```
scoping
```

`<project>/.keel/lifecycle.lock.toml`: a verbatim copy of the chosen lifecycle
TOML, frozen at `keel new` time.

## Migration path

`keel migrate` (existing) gains a v0.0.x → v0.1.0 step. For every project
under `PROJECTS_DIR`:

1. **Move manifests.** `design/project.toml` → `project.toml`,
   `design/milestones.toml` → `milestones.toml`,
   `design/.phase` → `.keel/phase`. Create `.keel/` if missing.
2. **Promote design files.** `design/scope.md` → `scope.md`,
   `design/design.md` → `design.md`. `design/decisions/` → `decisions/`,
   `design/plans/` → `plans/`, `design/specs/` → `specs/`.
3. **Convert deliverable.toml → project.toml.** Rename the file. Drop the
   `parent_project` field (path-derived now). Move `shared_worktree` from
   `[deliverable]` to `[project]`. Rename `[deliverable]` table to
   `[project]`.
4. **Snapshot lifecycle.** Read `<projects-dir>/.keel/lifecycles/<name>.toml`
   for the resolved lifecycle, copy verbatim to `<project>/.keel/lifecycle.lock.toml`.
5. **Generate README.md.** Render the template using current manifest +
   phase. Skip if a README already exists at the root.
6. **Update `Scope` and command code.** Path API in `keel.workspace.Scope`
   gains four new properties: `keel_dir`, `phase_path`, `lifecycle_snapshot_path`,
   plus updated `manifest_path` (now at root). The old `design_dir` becomes
   `human_design_root` or similar — pure-content directory, no manifests.
7. **`keel migrate` is idempotent.** Re-running on an already-migrated
   project is a no-op.

The migration script ships in `keel-cli 0.1.0`. Users on 0.0.x:

```bash
pip install --upgrade keel-cli
keel migrate --all
```

A blank-slate migration fallback: if a project is too far from the
expected layout (hand-edited oddities), the migrator prints a clear
"manual migration required: see <doc>" error and stops without writing.

  Multi-provider remains a future concern. Recorded in section "Non-goals".
- **#5 (`[extensions.ticketing]` selector pattern).** Documented as the
  canonical "selector + per-plugin sub-block" extension shape.
- **#6 (lifecycle = "default" implicit).** Frozen via `.keel/lifecycle.lock.toml`

## Future plugin space — `keel-ai` as architecture validation

A useful stress test for the redesigned plugin architecture is to imagine
a substantially different plugin and check whether it fits naturally.
`keel-ai` — a future plugin adding AI features scoped per project — is
the test case. Recording it here both as a parking-lot idea and as a
sanity-check on the decisions above.

### What `keel-ai` does (sketch)

A plugin that gives each keel project a local agent with full context of
its design corpus. Concretely:

- **RAG over project markdown.** `keel ai search "<query>"` returns
  ranked, cited chunks from `scope.md`, `design.md`, `decisions/`, and
  `plans/`. `keel ai chat` answers questions like "what blocks leaving
  the designing phase?" or "summarize our ticketing decisions."
- **Vector index lives per-project.** Embeddings cached under
  `.keel/ai/index/`, refreshed when files change.
- **Plugin tools and skills for project management.** The plugin ships
  a baseline set (search, summarize, draft-decision) and lets other
  plugins register their own tools that the agent can call (e.g.
  `keel-jira` exposes `create_ticket` as a tool the agent invokes when
  the user says "file this as a Jira ticket"). User-customizable via
  config; replaceable per project.
- **MCP server, project-scoped.** `keel ai mcp` serves the project's
  RAG + tools to external agents (Claude Code, Cursor) over MCP.

### Implementation hints (for whoever picks this up)

- **Chunking:** heading-aware markdown splits; respect frontmatter and
  decision-file structure.
- **Embeddings:** configurable provider. Default to **Voyage AI**
  (Anthropic-recommended) with OpenAI / Cohere / local Ollama as
  alternatives via `[extensions.ai] provider = "..."`.
- **Storage:** **MongoDB Atlas Local** as the suggested default —
  free, runs locally via Docker, supports vector search, and lets users
  graduate to Atlas Cloud without code changes. Other backends pluggable
  (LanceDB, Chroma, sqlite-vec) via the same selector pattern.
- **Hybrid ranking:** vector + keyword (BM25), fused with reciprocal
  rank fusion.

### What `keel-ai` needs from the architecture

| Need | Provided by | Status |
|---|---|---|
| Read all markdown in a project | Option-D layout (scope.md, design.md at root; decisions/, plans/ flat) | ✓ One less folder hop than today |
| Per-project tool state (embedding cache) | `.keel/` directory | ✓ `.keel/ai/index/` is the natural home |
| Per-project config | `[extensions.ai]` in `project.toml` | ✓ Pattern matches ticketing |
| Provider selection (Voyage / OpenAI / local) | `[extensions.ai] provider = "..."` | ✓ Same pattern as ticketing |
| File-change events ("re-embed when scope.md is edited") | None today | ⚠ New `keel.file_events` entry-point group needed |
| Tools other plugins can register for the agent | None today | ⚠ New `keel.agent_tools` entry-point group needed |
| MCP server scoped to a project | None today | ⚠ New `keel ai mcp` command; no architectural blocker |

### Architecture changes implied

The current redesign handles most of this. Three additions for whenever
`keel-ai` is built:

1. **`keel.file_events` entry-point group** — fires when keel writes or
   modifies project files. Lets re-indexers stay in sync without polling.
2. **`keel.agent_tools` entry-point group** — lets plugins register
   functions an agent can call. `keel-jira` could expose `create_ticket`
   as an agent tool; `keel-ai` consumes whatever's registered. This is
   plugin-of-plugins: keel-ai is a host for tools other plugins ship.
3. **`keel.skills` entry-point group** (optional, defer until needed) —
   higher-level skills (prompt + tools + policy) that wrap tools.

None of these block v0.1.0; they're follow-on work.

### Why noting this matters for the current spec

Stress-testing against `keel-ai` validates two of the four decisions:

- **`.keel/` for tool state** holds up perfectly — embedding caches,
  vector indexes, future MCP server state all fit. If state had stayed
  at the project root or flat, every plugin would invent its own
  convention for multi-megabyte caches.
- **Per-plugin config schemas + selector pattern** generalize beyond
  ticketing. AI providers (Voyage / OpenAI / local) follow the same
  shape as ticketing providers (Jira / GitHub / Linear). One mental
  model for users and plugin authors alike.

If `keel-ai` were impossible to express against this architecture, that
would be a sign the design is too ticketing-centric. It's not.

## Implementation scope (planned in a separate document)

The implementation plan will live at
`design/plans/<date>-data-model-redesign.md`, generated via the
`superpowers:writing-plans` skill once this spec is approved by the user.

Rough order of work:

1. New `project.toml` schema (drop deliverable models, add
   `shared_worktree` to ProjectMeta, generic `[extensions]` plumbing).
2. New `Scope` API (path properties point at the new layout).
3. `.keel/` directory creation in `keel new`; lifecycle snapshot.
4. `keel migrate` v0.0.x → v0.1.0 step.
5. Implicit-default milestone behavior in `keel task add`.
6. `README.md` template + `keel show --update-readme`.
7. Drop `deliverable.toml`, treat `keel deliverable add` as scaffolding
   sugar for nested project.
8. New `TicketProvider` protocol (typed objects, no `parent_id` parameter).
9. keel-jira v0.1.0 — refactor to the new protocol, add per-plugin
    templates with sensible defaults. (Plugin picks its own templating
    engine; no shared helper in keel-cli core.)
10. CONTRIBUTING.md — document `[extensions]` conventions, plugin
    extension points, migration notes.
11. README + design.md — refresh to reflect the new layout.

Estimated size: comparable to Plan 7 (customizable lifecycles), maybe
slightly larger because the migration touches every command. ~15-20
implementation tasks.

Both wheels jump to 0.1.0 on completion.
