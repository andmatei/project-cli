# Plan 4.5 cleanup before Plan 5 (milestones + plugins)

## Summary

After Plans 1-4 shipped, three review agents (architecture, UX, plugin-readiness)
identified a punch list of items worth landing before Plan 5 (milestones + a
plugin system for ticketing integrations). Plan 4.5 implements the high-leverage
items; this decision records what changed and what's intentionally deferred.

## What changed

### Plugin-readiness scaffolding (M1)

- **`extensions: dict[str, Any]`** field added to `ProjectManifest` and
  `DeliverableManifest`. `extra="forbid"` is preserved on every other field.
  Plugins put their config under `[extensions.<namespace>.<plugin>]`, e.g.
  `[extensions.ticketing.jira]`. Round-tripped via `tomlkit`.
- **`keel.testing`** package now hosts `projects`, `make_project`,
  `make_deliverable`, `source_repo` fixtures. Plugin authors enable them with
  `pytest_plugins = ["keel.testing"]` in their conftest.
- **`keel.api`** module re-exports the public/stable surface. Anything outside
  `keel.api` and `keel.testing` is internal and may change between minor
  releases.
- **`app.py`** loads command plugins via `[project.entry-points."keel.commands"]`.
  An empty `ticketing` Typer subapp is pre-registered for Plan 5's bundled and
  third-party providers to extend.

### Path/scope abstraction (M2)

- **`Scope` dataclass** promoted from data-only to behavioral. Properties:
  `unit_dir`, `design_dir`, `manifest_path`, `phase_file`, `decisions_dir`.
- **`workspace.design_dir`, `manifest_path`, `phase_file`** helpers added.
- **17 command files swept** to use the new helpers. ~90 hardcoded
  `design/<file>` literals removed.
- **`keel.lifecycle`** module created hosting `PHASES`, `DEFAULT_PHASE`,
  `next_phase()`, `is_valid_phase()`. `phase.py` and `workspace.read_phase`
  import from there. Plan 5's customisable lifecycle DSL will plug into the
  same module.

### UX consistency (M3)

- Duplicate `Created project: ...` (stderr) line removed from `new`,
  `deliverable add`, `decision new`, `phase` (transition mode).
- `validate` summary line moved to stdout (was on stderr while the table was
  on stdout — broke pipelines).
- Fix-it hints appended to all "not found" errors. Examples:
  `keel show ghost` → `error: project not found: ghost\n  Hint: run 'keel list' ...`
- README section added documenting project-name resolution rules
  (positional vs `--project` vs CWD detection).

### Polish (M4)

- **`keel.errors.ErrorCode`** enum: standardises the ~30 `code=...` strings
  used in `out.error` calls. Stable contract for JSON consumers.
- `workspace.resolve_cli_scope` now routes errors through `Output.error`
  (was raw `typer.echo`, which lost the JSON `code` field).
- Function-local imports hoisted to module-level where no circular-import
  justification exists.

## What's intentionally deferred

These UX findings are real but don't compound with Plan 5; they'll land as a
future polish pass.

- **`phase` ↔ `show` panel consistency**: `phase` is plain text, `show` is a
  Rich panel. Cosmetic; doesn't affect Plan 5.
- **`design export` double-H1**: the export wrapper injects `# <project>` then
  the design.md begins with `# <project> — Design`. Fix in a polish pass.
- **`-y` flag semantics**: `keel new beta -y` still requires `-d "..."`. The
  flag means "skip confirmation prompts" not "default missing args". Document
  rather than redefine.
- **JSON schema unification across `list`/`show`/`phase`**: the shapes are
  inconsistent; unifying is its own design exercise (likely Plan 6).
- **Decision rm vs supersedes docs**: the help text could be sharper about
  the audit-trail tradeoff.
- **Phase transition discoverability**: `keel phase` (read mode) doesn't list
  valid next phases. Small UX gap; not blocking.
- **Markdown_edit unused helpers**: `section_exists`, `replace_section`,
  `remove_line_under_heading` are test-only. Decide later: delete or
  document as plugin-facing API.

## Consequences

- `keel.api` becomes the stable plugin contract. Bumping the minor version
  with breaking changes to `keel.api` requires the next major. Things outside
  `keel.api` are free to evolve.
- `Scope` is now opinionated (carries path-resolution behavior). Tests that
  construct `Scope` directly should keep working — the new properties are
  additive.
- The `extensions` manifest field is the only "open" field in the schema.
  Plugins MUST namespace their data; leaking config into top-level keys still
  fails with `extra="forbid"`.
- The `[project.entry-points."keel.commands"]` and
  `[project.entry-points."keel.ticket_providers"]` groups are now reserved.
  External plugins should target these.
